# OpenClass 工作链路流程框架文档

## 1. 文档目的

本文描述 OpenClass 的工作链路流程框架，用于说明用户输入如何被系统转化为可定位、可执行、可追溯的文档工作。

本文不是 PRD（产品需求文档，Product Requirements Document），也不是代码实现说明。PRD 负责定义产品目标、范围和验收标准；本文负责说明 OpenClass 的产品工作链路、角色边界、任务拆分、资料证据流转和文档变更流程。

## 2. 总体工作链路

OpenClass 的核心链路是：

```text
用户输入
-> 判断本轮意图
-> 判断文档状态和目标位置
-> 形成任务清单
-> 选择资料证据
-> 生成讲解指令或文档变更计划
-> BoardEditor 执行文档生成、扩写、编辑或删除
-> Chatbot 承接结果
-> 写入历史和 metadata
```

这里的核心不是让 Chatbot（聊天机器人）直接回答一切，而是把用户的自然语言请求转化成围绕 Board（右侧文档）的结构化工作。

## 3. Resource Pipeline（资料处理流水线）

用户一开始可能上传文件资料。资料链路不应该被设计成一个万能的“文件资料管理 AI”，而应拆成四个模块。

### 3.1 Resource Library（资料库）

Resource Library 负责接收用户上传的文件，保存原文件、文件 metadata（元数据）、解析结果、章节结构、文本片段和必要索引。

它只负责让资料变成可查、可定位、可引用的资源，不负责决定本轮是否使用资料。

### 3.2 Resource Indexer（资料索引器）

Resource Indexer 负责把上传资料拆成可检索的结构，例如章节、页码、标题路径、正文片段、图表或公式位置。

它生成的是通用资料结构，不写入具体学科、教材、考试或 demo（演示样例）规则。

### 3.3 ResourceResolver（资料证据选择器）

当用户的学习或文档需求已经比较清楚后，ResourceResolver 根据本轮需求去资料库中检索相关内容，判断是否有可用证据。

如果找到相关资料，它只返回命中的章节、位置、概览和置信度，不直接替用户决定必须使用资料。

### 3.4 Evidence Handoff（证据交接）

如果系统认为资料可能有帮助，就把命中的资料位置、章节概览和引用理由展示给用户，询问是否参考这些资料来生成或扩写 Board。

用户确认后，系统形成 Evidence Bundle（证据包），并交给 BoardEditor（文档编辑 AI）生成或修改文档。用户拒绝时，BoardEditor 只根据用户原始需求生成，不注入这批资料证据。

资料使用流程：

```text
用户上传资料
-> Resource Library 保存原文件和 metadata
-> Resource Indexer 解析章节、片段和索引
-> 用户提出学习 / 练习 / 写作需求
-> 需求清晰后，ResourceResolver 检索相关资料
-> 如果没有相关资料：只根据用户需求继续生成或扩写文档
-> 如果有相关资料：向用户展示命中章节、位置和概览
-> 用户确认参考：形成 Evidence Bundle，交给 BoardEditor
-> 用户拒绝参考：本轮不注入资料证据
-> 生成、扩写或编辑后的文档记录资料来源线索
```

## 4. Turn Intent（本轮意图判断）

系统首先判断用户在聊天框中是在普通聊天，还是有学习或文档工作需求。

### 4.1 ordinary_chat（普通聊天）

如果用户没有学习或文档工作需求，Chatbot 可以自然聊天。

普通聊天必须满足以下边界：

- 不能更新 LearningRequirementSheet（学习需求清单）。
- 不能读取右侧全文摘要。
- 不能误入 BoardTask（板书任务）链路。

### 4.2 learning_work（学习工作需求）

如果用户有学习、练习、生成、讲解、扩写、编辑、删除或互动需求，系统进入文档工作链路，并继续判断文档状态和目标位置。

### 4.3 unknown / narrow_topic（意图不清 / 主题过宽）

如果用户想学的内容表达不清，或者表达清楚但主题过宽，Chatbot 应在聊天框中进行引导和内容推荐。

此时系统可以追问一个关键问题，或给出几个内容主题方向，但不直接生成文档。

## 5. Target Resolution（目标解析）

系统需要判断当前文档中是否存在用户需求指向的内容，以及该内容是否能被稳定定位。

### 5.1 board_empty（板书为空）

如果 Board 为空，系统走首次从零生成文档链路。

### 5.2 target_resolved（目标已解析）

如果 Board 中能稳定找到目标，系统会把目标表示成两种文档目标之一：

- TargetRange（目标文本范围）：要对哪一段已有内容进行动作。
- InsertionAnchor（插入锚点）：新内容要插入到哪个位置。

目标解析完成后，系统进入 BoardTaskRequirementSheet（板书任务清单）链路。

### 5.3 target_not_resolved（目标未解析）

如果系统没有稳定定位到用户指向的内容，不能直接判断内容不存在，也不能直接执行讲解、编辑或扩写。

target_not_resolved 可能来自以下原因：

- 位置不清。
- 用户问法和文档表达不匹配。
- 检索漏掉。
- 当前文档确实没有相关内容。

此时系统只能继续定位，或询问用户是否把该内容作为新内容扩写进文档。

系统可以通过文段相关性猜测候选位置，并推荐适合扩写的位置，但不能把“没找到”直接当成“文档没有”。

## 6. Initial Document Generation（首次文档生成）

当 Board 为空时，系统需要判断用户是要学习某个知识点，还是要生成练习材料。

### 6.1 学某知识点

如果用户要学某个知识点，系统需要先把学习主题细化到足够明确的知识点。

主题明确后，系统走资料证据选择链路，并通过扩写动作生成知识点讲解文档。

生成后，Chatbot 只询问用户是否需要从头讲解新生成内容。用户确认后，系统再按小节进行讲解。

### 6.2 做相关练习

如果用户要做某项相关练习，系统需要维护练习需求清单。

清单至少包含：

- 用户要练习什么内容。
- 用户当前相关领域水平。
- 用户练习相关内容的目的。

清单填写完善前，Chatbot 只追问关键缺口，不主动进入下一步工作。除非用户明确要求直接生成，系统才可以在当前信息下推进。

## 7. BoardTaskRequirementSheet（板书任务清单）

当 Board 不为空，并且目标已解析后，Chatbot 根据用户透露的信息填写 BoardTaskRequirementSheet。

清单固定收集四项：

- 目标位置。
- 动作类型。
- 问题 / 主题。
- 是否有特殊互动要求。

清单完善后，系统再交给板书侧角色执行。

### 7.1 explain（讲解）

如果用户要讲解 TargetRange，讲解路线必须先由 Board AI 生成 BoardExplanationDirective（板书讲解指令）。

BoardExplanationDirective 应包含：

- 目标摘录。
- 讲解边界。
- 是否允许讲解。
- 给 Chatbot 的讲解指令。

Chatbot 只能根据该 directive 讲解，不能凭常识直接讲右侧文档。

### 7.2 mutate_board（文档变更）

如果用户要对文档进行编写、扩写、改写或删除，系统进入 BoardMutationPlan（板书变更计划）链路。

BoardMutationPlan 负责把用户一句自然语言请求拆成文档动作清单：

- edit：改写已有内容。
- write：新增内容。
- delete：删除内容，属于高风险动作，需要确认。

例如用户说：“把这一节改得更通俗一点，再补一个例子。”

系统应拆成两个文档动作：

1. edit：改写已有内容。
2. write：新增例子。

拆分后，系统进入 BoardMutationPlan 执行链路。

### 7.3 interaction（规则互动）

当用户表达有特殊互动方式要求时，系统可以围绕特定 TargetRange 启动规则互动。

系统根据用户的特殊交互方式要求，在 prompt（提示词）中定义互动规则，说明 AI 和用户之间如何循环交互。

互动过程中：

- 用户输入合规时，系统继续按规则互动。
- 用户输入不合规时，可以先 rule_violation（规则内纠错），不一定立刻退出。
- 用户明确结束时，进入 exit_rule。
- 用户提出新任务时，进入 new_task，并把用户输入重新送回总链路判断。

## 8. BoardMutationPlan（板书变更计划）

BoardMutationPlan 负责具体规划如何修改 Board。它不是直接写内容的角色，而是把自然语言请求变成结构化文档动作。

### 8.1 edit（编辑已有内容）

edit 用于修改 TargetRange 中已有的内容。

系统根据 BoardMutationPlan 修改目标内容，并写入版本历史。

编辑完成后，Chatbot 只说明已完成修改，并询问用户是否需要讲解本次改动或新版本内容。

如果用户确认需要讲解，再进入 BoardExplanationDirective 链路。

### 8.2 write / expand（写入 / 扩写新增内容）

write / expand 用于在 InsertionAnchor 附近新增内容。

系统先确认或解析合适的 InsertionAnchor，再进入 ExpansionPlan（扩写计划）链路，决定新增内容的篇幅尺度和结构。

### 8.3 delete（删除已有内容）

delete 用于删除 TargetRange 中已有内容，属于高风险文档变更。

删除动作必须要求用户确认，并在 commit metadata（提交元数据）中记录删除范围、删除原因和执行结果。

## 9. ExpansionPlan（扩写计划）

ExpansionPlan 负责决定扩写时写多大、写到哪里、使用什么结构。

### 9.1 从零生成板书

如果 Board 为空，从零生成通常使用 whole_board（整篇板书级）尺度。

### 9.2 已有板书内扩写

如果 Board 已经有内容，系统根据新内容和当前文档主题的关系决定扩写方式：

- 属于当前文档主题：在相关父章节下新增。
- 是独立主题：询问用户创建新 lesson，还是追加到当前文档。
- 用户明确要完整新板书：使用 whole_board 尺度生成。

### 9.3 篇幅尺度分类

ExpansionPlan 可以使用以下篇幅尺度：

- sentence（句子级）：补一句。
- paragraph（段落级）：补一段。
- micro_section（小节级）：补一个小标题下的小块内容。
- section（章节级）：补一个完整章节。
- whole_board（整篇板书级）：生成一整篇板书。
- lesson_set（多个 lesson 级）：生成一组学习单元。

## 10. History / Metadata（历史与元数据）

所有生成、编辑、扩写、删除、互动启动、讲解授权，都必须留下 run / version / commit metadata（运行记录 / 版本记录 / 提交元数据）。

历史记录至少应能说明：

- 本轮用户意图是什么。
- 使用了哪个目标范围或插入锚点。
- 是否使用资料证据。
- 执行了哪些文档动作。
- 是否生成 BoardExplanationDirective。
- 是否启动 InteractionSession（互动会话）。
- 是否消费了对应任务清单。

历史与元数据是 OpenClass 可追溯文档工作台的一部分，不是实现细节。
