# OpenClass Review Snapshot — 2026-05-23

This is an old project-health snapshot. Treat it as context, not current truth; verify against the repo before acting.

## Main Findings

1. `course-studio.tsx` had become the largest maintenance risk.
   - It mixed editor configuration, page settings, toolbar, DOCX import/export, version history, AI chat, graph UI, model selection, and many state/effect paths.
   - Direction: keep it as a composition shell and move behavior into child components, hooks, and `src/lib/`.

2. Service-layer boundaries were mostly good, but store access and SQL ownership needed tightening.
   - Direction: keep router logic thin, reuse store/service transaction helpers, and avoid new scattered SQL.

3. Some schema looked ahead of runtime behavior.
   - Direction: distinguish active API behavior from future workflow schema so future agents do not assume unfinished teaching flows are fully wired.

4. AI workflow was in a transition state.
   - Direction: reconnect AI through the explicit OpenClass workflow in `AGENTS.md`, not through fallback templates or subject-specific shortcuts.

5. The SQLite model was pragmatic but has scaling limits.
   - Direction: keep single-writer constraints explicit, preserve WAL and backups, and defer block-level storage until the product needs it.

6. Test coverage was stronger on backend than frontend.
   - Direction: protect large refactors with focused fixtures and Playwright smoke coverage.

## Strengths

- Strict Pydantic modeling.
- Multi-provider model catalog.
- Lesson commit / branch history.
- Resource parsing and chapter extraction boundaries.
- Real deployment notes and local run scripts.

## Working Rule

When this review conflicts with current code or current `AGENTS.md`, current code and `AGENTS.md` win. Use this document only to remember why the project keeps pushing toward smaller modules, explicit workflow state, and generic AI capabilities.
