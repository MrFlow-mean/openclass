# 锐评 OpenClass：一个野心很大、执行半程、债务开始堆积的课程工作台

## 总体印象

这不是一个"小工具"，而是一个**试图用精密的架构图纸去约束混乱的工程现实**的项目。文档（`AGENTS.md`、`README.md`）写得像宪法，代码里却已经开始出现违宪行为。

---

## 一、前端：5031 行的 `course-studio.tsx` —— 这是你最大的技术债

一个文件，5031 行，集齐了：
- TipTap 编辑器配置
- 页面设置面板
- 工具栏
- DOCX 导入导出
- 版本历史/分支 UI
- AI 聊天侧边栏
- 课程图谱相关逻辑
- 模型选择
- 各种 useState/useEffect

**`AGENTS.md` 第 62 行明确写着**：
> `course-studio.tsx`：只做顶层组合，不继续堆状态、effect、realtime、editor、model selection 逻辑。

**现实是**：你不仅堆了，还堆了 5000 多行。这不是"顶层组合"，这是**前端大泥球**。一旦需要加一个新功能（比如协作光标、评论批注），维护者会在这个文件里迷路。

**建议**：立刻拆分。编辑器核心、页面设置、AI 侧边栏、历史面板、工具栏至少应该是 5 个独立组件/模块。

---

## 二、后端：架构分层不错，但"全局单例"和"SQL 散弹枪"在腐蚀它

### 2.1 `workspace_state.py` 里的全局变量

```python
STORE = SqliteCourseStore(DATABASE_PATH, legacy_json_path=LEGACY_STORE_PATH)
```

一个模块级全局 SQLite 连接池替代物。FastAPI 是 async 框架，虽然你用的是 sync SQLite + threading.RLock，但这意味着：
- 无法 mock 测试（除非 monkeypatch 模块全局变量）
- 无法做读写分离或连接池调优
- 单元测试之间会互相污染

更现代的做法应该是依赖注入（FastAPI `Depends`），把 `CourseStore` 作为可替换的接口传入。

### 2.2 `auth_service.py`：Service 层自己在拼 SQL

`AGENTS.md` 说：
> 不要在 router 里直接拼 SQL 或绕过 service 事务。

但它没说**service 层可以随便拼 SQL**。`auth_service.py` 里有大量的 `INSERT INTO users ...`、`SELECT ... FROM auth_identities`，而且用的是和 `course_store.py` **不同的连接逻辑**。这意味着：
- 事务边界不清晰。如果一次操作既要改课程又要改用户信息，是两个独立事务。
- schema 变更时需要改多处。

**auth 应该要么走统一 store，要么有一个独立的 `AuthStore` 类，而不是在 service 里裸写 SQL。**

---

## 三、数据模型：Pydantic 用得很漂亮，但也过度设计了

`models.py` 是一个 impressive 的类型定义文件，定义了：
- `BoardSegment`, `BoardSegmentIndex`, `BoardFocusRef`
- `BoardTeachingGuide`, `BoardTeachingProgress`, `SectionTeachingProgressView`
- `ResourceReferencePrompt`, `ResourceContextChunk`
- `RealtimeConnectRequest`, `GoogleRealtimeSessionRequest`

问题是：**其中一大半模型，当前的 API 根本不返回或不做任何有意义的事。**

`chat.py` 和 `realtime.py` 已经返回"已移除"状态。那些精美的 `BoardTeachingGuide`、`BoardTeachingProgress`、`SectionTeachingPlan` 就像是一栋精装修的样板房，但整栋楼目前只有毛坯在使用。

**技术债形式**：未来的开发者（或 AI）会被这些模型误导，以为系统支持完整的 AI 教学流程，实际上大部分只是 schema 层面的占位符。

---

## 四、AI 能力：现在是"断臂维纳斯"

README 坦诚地说：
> 旧的后端 AI 工作流程运行框架已经移除，新的产品工作架构等待重新接入。

这意味着当前 `build_lesson_for_topic` 走的不是真正的 LLM 生成，而是 `lesson_factory.py` 里的硬编码 fallback。一个 AI 课程工作台，**核心 AI 生成能力是断掉的**。

这本身不是问题（重构期很正常），但问题是你保留了一整套**为完整 AI 工作流设计的 schema 和 service 空壳**。这些空壳每天都在增加理解成本。

---

## 五、存储模型：JSON 嵌套大对象 vs 关系型数据

你把 `CoursePackage` → `Lesson` → `BoardDocument` → `content_json` 全部作为嵌套 JSON 塞进 SQLite。好处是简单，坏处是：

- **无法高效查询**：想找"所有包含公式块的文档"，需要把全表读出来反序列化。
- **无法做增量更新**：改一个段落也要重写整个 lesson 的 JSON。
- **版本历史冗余**：每次 commit 存完整的 `BoardDocument` 快照，长期运行后数据量会膨胀。

对于当前规模（个人/小团队备课工具），这够用。但如果未来想支持"搜索所有课程中的某段文本"或"协作编辑"，这套存储模型会是第一道坎。

---

## 六、测试：3332 行测试 vs 整个系统，前端零测试

后端测试覆盖了一些关键点（store、auth、workspace router、history），但：
- 没有**集成测试**验证端到端的工作流
- 没有**前端测试**（没有 jest/vitest/playwright 配置）
- `course-studio.tsx` 那种 5000 行的组件，没有测试就等于没有重构安全网

---

## 七、部署：SQLite + Docker + 单进程，这是"自知之明"还是"自我限制"？

文档写得很好：
> 拓扑约束：单后端写入进程 + 文件级备份 + WAL；不允许多机/多进程同时写同一 sqlite。

这很务实——承认 SQLite 的局限并给出约束。但如果未来用户量上去，或者需要零停机部署，这套架构会立刻成为瓶颈。这不是批评，只是指出：**你现在的架构天花板很低**，低到有明确的上界。

---

## 八、`AGENTS.md`：写得最好的文件，也是最大的讽刺

这份 AI 协作指南是我见过的**最严谨的此类文档之一**。它有：
- 明确的"宪法"原则（通用能力优先）
- 文件边界定义
- 自检表
- 严禁事项清单

**但代码已经在违反它**了：
- `course-studio.tsx` 堆了 5000 行状态逻辑 ❌
- `fallback_generator.py` / `lesson_factory.py` 里有一些接近硬编码的默认文本 ❌（虽然不算学科硬编码，但离"领域无关"有距离）
- `openai_course_ai.py` 里直接加载 `.env`（`Path(__file__).resolve().parents[4] / ".env"`）——路径魔法数 ❌

**文档越是完美，与现实的落差就越刺眼。**

---

## 九、亮点（锐评也得讲好的）

1. **Pydantic 建模非常规范**：类型严格、默认值合理、`model_validator` 用得好（比如 `upgrade_legacy_blocks`）。
2. **多 Provider AI 抽象**：`ai_model_catalog.py` 的设计是成熟的，支持 7+ 个 provider 且前端能动态显示配置状态。
3. **版本历史设计**：Commit + Branch 的模型，比市面上多数内容工具做得深。
4. **资料库与章节解析**：`resource_library.py` + `resource_service.py` 的分层和 PDF/DOCX 解析流程是合理的。
5. **部署文档详尽**：从本地 `npm run setup` 到线上 `ssh + docker compose` 的每一步都写了，说明确实在真实环境跑过。

## 一句话锐评

> **OpenClass 像是一个读了大量软件工程经典、写了一份完美架构宪法的团队，然后在 deadline 压力下，把前端所有逻辑塞进了一个 5000 行的 React 组件里。它有成为优秀产品的骨架，但现在最需要的是一把手术刀——先把 `course-studio.tsx` 大卸八块，再决定 AI 工作流怎么重新长回来。**
