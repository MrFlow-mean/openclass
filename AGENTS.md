# OpenClass — AI 协作宪法

OpenClass 是面向严肃学习、研究、写作和知识工作的 AI document workspace（AI 文档工作台）。
它不是学科模板系统，也不是无边界的 general agent（通用智能代理）。

本文件是仓库根目录的强制工程约束。目录内存在更近的 `AGENTS.md` 时，遵循其补充规则；它不得削弱本文件的宪法级边界。

## 0. 最高优先级：保持系统小、清楚、可演进

**不要为了完成当前一句需求，让项目新增一层长期复杂性。**

每次改动按以下优先级选择方案：

1. 删除已经失效或重复的逻辑。
2. 复用已有的状态对象、服务、接口和测试。
3. 在已有明确边界中补齐能力。
4. 仅在现有边界无法表达时，抽取小而独立的模块。
5. 只有产品确实需要长期新能力时，才新增依赖、表、服务、流程或抽象层。

禁止：

- 为单一案例新增第二套 workflow（工作流）、数据模型、资料管线或 UI（用户界面）。
- 用 feature flag（功能开关）、兼容分支、临时 fallback（兜底）无限保留旧路径；必须有删除条件和迁移结束点。
- 在一次任务中顺带重构无关模块、更新依赖、重做 UI 或改变部署。
- 新增“manager / coordinator / helper”一类没有独立状态、权限或生命周期的空壳层。
- 让同一事实同时存于多个 state（状态）对象、多个 API（接口）字段或平行资料索引中。
- 把一次模型输出、一次用户话语或一个测试样例固化成永久业务规则。

修改前必须先给出简短的设计判断：

```text
问题属于：通用能力 / 内容形态 / UI 交互 / prompt 质量 / schema / 资料解析 / adapter / 测试
现有可复用边界：
最小改动文件：
本次不改什么：
是否新增依赖、持久化字段、后台任务或并行路径：否 / 是（说明必要性和退出条件）
```

若改动跨越多个层、需要超过合理的最小文件集，先说明拆分方案；不要用大 PR（合并请求）把不相关问题绑在一起。

## 1. 通用能力优先

核心代码只处理通用学习能力、内容形态、资料结构、用户意图、文档操作和模型调用。

不得在默认核心路径加入：

- 学科、教材、考试、语法点或具体案例关键词分支。
- 固定讲义 HTML（超文本标记语言）、固定课程模板、demo 内容。
- 针对单一资料、单一句用户输入或单个测试样例的特殊逻辑。

允许的抽象包括 `request.intent`、`document_shape`、`user_goal`、结构化资料特征和明确的文档范围。若能力只适用于某个外部系统或文件格式，优先放进 plugin / adapter（插件 / 适配器），而不是污染默认路径。

用户可见的 AI 回复由模型根据上下文生成。不要在核心路径添加套话式固定回复、固定讲解正文或为演示而设的默认内容。

## 2. 正式 Board 变更的受控执行边界

`Board` 是用户的正式文档，不是聊天草稿。

- `Chatbot` 可以理解需求、解释、提出建议，并生成 `BoardMutationPlan`（文档变更计划）。
- 正式 Board 的生成、写入、编辑、替换或扩写，必须通过 `DocumentMutationExecutor`（文档变更执行器）。当前实现可由 `BoardEditor` 或同一模型承担；关键是执行权受控，而不是必须另起一个模型。
- 执行前必须具备适用的目标范围、变更意图、资料证据和用户确认；目标不明确时先澄清，当前文档没有目标内容时先确认是否扩写。
- 执行后必须写入可追溯的版本 / commit（提交记录）与 metadata（元数据）。Chatbot 不得绕过执行边界，也不得宣称文档已变更。
- 不要把“提出计划”和“提交正式变更”合并为一个无法审计的聊天回复。

## 3. 文档、资料与审计事实来源

- `content_text` 是正式文档的事实来源，可使用 Markdown（轻量标记格式）、普通文本或 LaTeX（数学排版语言）。
- `content_html`、HTML DOM（浏览器文档对象模型）和编辑器节点只属于展示或导出层，不得反向取代正式事实。
- 资料引用必须保留 `source_ingestion_id`、`source_chapter_id`、页码 / 范围 / chunk（文本块）身份及确认、冻结、消费状态；不得退化为无来源 prompt 文本。
- 资料原件、解析结构、证据选择和下游交接是不同阶段；不要把它们混进一个大服务或一次聊天调用。
- 任何真实 Board 变更、冻结、确认、消费、失败或取消都应保留历史痕迹；普通聊天不应伪造文档或任务历史。

## 4. 工作流与职责边界

先判断 Board 是否为空：

- 空白 Board：收集最小 `LearningRequirementSheet`（学习需求清单），满足条件后冻结，再生成正式文档。
- 已有 Board：使用 `BoardTaskRequirementSheet`（板书任务清单）明确目标、动作、主题和特殊互动要求；通过 `FocusResolver`（目标定位器）后再执行 `write`、`edit`、`explain` 或 `chat`。

职责保持单一：

- router（接口路由）只处理 HTTP（网络接口）边界和输入输出转换，不直接拼 SQL（数据库查询语言）或承载业务事务。
- service（服务层）拥有业务规则、事务和状态转换。
- `Chatbot` 是对话与编排入口，不直接吞入文档执行、资料定位或所有新业务规则。
- `BoardEditor` / 执行器拥有正式文档变更。
- `FocusResolver` 只定位；不可靠时返回澄清，而不是猜测。
- `ResourceResolver` 只选择相关证据；不默认污染所有模型上下文。
- `InteractionSession` 只管理明确规则的互动，不把临时对话无限写成持久状态。

不要让 `chatbot.py`、`openai_course_ai.py`、`course-studio.tsx`、`word-board-editor.tsx` 或 `rich_document.py` 继续吸收不属于其职责的新逻辑。若确实需要改动这些文件，先说明为什么现有边界无法承接，并同时避免把无关逻辑带入。

## 5. 前端、后端与数据边界

- 前端组件通过 `apps/web/src/lib/` 的 API client（接口客户端）请求后端；组件层不直接 `fetch`。
- 视图类型放在 `apps/web/src/types/`，与后端 API 视图一致；不得把本地文件路径或资料原文暴露到前端。
- `course-studio.tsx` 只组合界面；可复用状态和副作用进入 `hooks/` 或小组件。
- 后端写入经 service 层和现有事务路径；不要在 router 中绕过存储边界。
- SQLite（轻量关系型数据库）、上传文件、日志和导出物是运行数据，不得提交；不要让多个独立写进程同时写同一数据库。

## 6. 验证、文档与交付

按风险运行最小充分验证，而不是机械运行无关任务：

```bash
npm run guard:file-sizes
npm run lint:web
npm run typecheck:web
npm run test:api
npm run build:web
npm run test:e2e
npm run verify
```

- 涉及自然语言路由、定位或资料选择时，补充正例与至少两个反例；测试通用信号，不测试某个学科案例。
- 修改公开行为、持久化数据或工作流时，更新相应文档与测试；不要把架构长文复制进 README（项目入口说明）。
- 开始时先检查 `git status`，只暂存当前任务文件，保留其他人的未提交改动。
- 每次代码或仓库文档改动默认提交并同步 GitHub；仅在用户明确要求讨论、草稿或不提交时例外。不得未经明确授权 force push（强制推送）、重写历史或改动生产数据。

完成时报告：改动文件、为何没有引入臃肿或领域硬编码、保留的既有链路、实际验证结果，以及仍未验证的风险。
